¿En qué se diferencia crear una tabla en SQL de crear una colección en MongoDB? Piensa en términos de rigidez estructural, necesidad de una declaración previa de columnas y cómo actúan los tipos de datos en la inserción inicial.

1. El Concepto de Base de Datos en MongoDB

En MongoDB, una base de datos actúa como el contenedor físico-lógico de nivel superior para las colecciones de datos. Al igual que en los ecosistemas relacionales tradicionales gobernados por estándares SQL, cada base de datos en MongoDB posee su propio conjunto exclusivo de archivos en el sistema de almacenamiento físico. Sin embargo, la diferencia radical yace en el ciclo de vida de la instanciación: en MongoDB, una base de datos no se crea realmente hasta que no se le inserta su primer documento.

"La filosofía de diseño de MongoDB favorece la asignación dinámica sobre la marcha de esquemas y descriptores lógicos de almacenamiento, optimizando la agilidad de los entornos de desarrollo ágiles."
— Manual Oficial de Referencia de MongoDB Arquitectura.

2. Taxonomía de las Colecciones: Flexibles vs Fijas (Capped)

A diferencia de SQL, donde todas las tablas operan bajo principios de crecimiento físico elástico limitado por el disco duro, MongoDB divide sus agrupaciones documentales en dos categorías principales:

Colecciones Flexibles (Estándar)

Son el tipo por defecto en MongoDB. No tienen restricciones de tamaño máximo ni límites explícitos sobre el volumen global de registros. Su crecimiento es dinámico y vertical. No requieren definición estricta de atributos estructurales previos a la inserción masiva.

Colecciones Fijas (Capped Collections)

Colecciones de tamaño circular preasignado que preservan de manera nativa el orden de inserción de los elementos. Al llenarse por completo el espacio en bytes parametrizado, el motor elimina automáticamente los documentos más antiguos para dar paso a los nuevos (comportamiento FIFO). Son ideales para logs o arquitecturas de telemetría temporal.

3. Validación de Esquemas (Schema Validation)

Aunque MongoDB se enorgullece de su diseño nativo libre de esquemas (*schemaless*), los entornos empresariales e industriales requieren mantener la calidad y consistencia e invariabilidad de los datos crudos. Para solucionar esta dicotomía, MongoDB implementa la Validación de Esquemas utilizando la sintaxis estándar de JSON Schema. Esto nos permite definir de forma programática campos obligatorios, tipos de datos rigurosos y rangos condicionales.

¿Cuándo aplicar validación de esquemas?

Se debe aplicar en colecciones core de negocio (como Usuarios, Finanzas, o Transacciones) donde la mutabilidad errónea de tipos (por ejemplo, guardar un número telefónico como entero en vez de string) rompería capas críticas del backend o aplicaciones cliente.

4. Las Reglas de Validación a Detalle

¿Qué son y cómo funcionan?

Las reglas de validación son directrices o restricciones de integridad que se configuran a nivel de colección. Cuando un cliente intenta guardar (insert) o modificar (update) un documento, el motor de MongoDB intercepta el documento JSON y lo evalúa contra el validador configurado **antes** de escribir en el almacenamiento físico.

Si el documento cumple con todas las reglas, se escribe con éxito; si viola alguna condición (por ejemplo, falta un campo obligatorio o el tipo de dato es erróneo), MongoDB **rechaza por completo** la operación devolviendo un error de validación (WriteError: Document failed validation).

¿Cómo crearlas?

Se definen mediante el operador $jsonSchema dentro del método de creación de la colección. A continuación, se muestra la estructura sintáctica estándar para inicializar reglas de validación rígidas: 

db.createCollection("Usuarios", {
   validator: {
      $jsonSchema: {
         bsonType: "object",
         required: [ "nombre", "correo" ],
         properties: {
            nombre: {
               bsonType: "string",
               description: "El nombre debe ser una cadena de caracteres y es obligatorio"
            },
            correo: {
               bsonType: "string",
               pattern: "^.+@.+$",
               description: "Debe ser un correo electrónico válido"
            }
         }
      }
   },
   validationLevel: "strict",
   validationAction: "error"
})

*Nota: validationAction: "error" le indica a MongoDB que bloquee la inserción. Si se cambia a "warn", grabará el registro pero dejará una alerta en el archivo log del servidor.*

5. Guía Rápida de Comandos (MongoDB Shell)

A continuación se enumeran los comandos de manipulación estructural:

Comando Shell Función Operativa Exacta
use <nombre_db> Cambia el contexto de ejecución actual a la BD indicada. Si no existe, la crea virtualmente.
db.createCollection("<nombre>", {opciones}) Crea explícitamente una colección permitiendo inyectar configuraciones de validación o límites.
db.dropDatabase() Elimina de forma irreversible la base de datos que se encuentra actualmente en uso activo.

Indicaciones Paso a Paso para los Equipos:
  1. Instanciación: Acceder a la CLI de Atlas o Compas local e instanciar una base de datos denominada de forma unificada como BibliotecaDigital.
  2. Colección Flexible: Generar la colección Libros sin aplicar reglas de validación. Insertar 5 documentos con campos totalmente heterogéneos para constatar la naturaleza flexible de la plataforma.
  3. Colección con Validación Estricta: Crear la colección Usuarios configurando obligatoriedad explícita para los atributos: nombre (string) y correo (string). Validar el comportamiento provocando un rechazo intencional del sistema mediante inserciones mal formadas.
  4. Colección Circular: Crear una colección fija llamada Prestamos limitada estrictamente a 1MB de espacio físico (`size: 1048576`) para retener exclusivamente logs transaccionales volátiles.
  5. Modificación Dinámica en Vivo: Realizar de forma grupal una alteración al validador de la colección de Usuarios para incorporar de manera obligatoria el nuevo campo telefono.
